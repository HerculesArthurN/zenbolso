# Plan 01-03: Desktop Interaction & UX Fixes

## Objective
Audit and fix all buttons, hover states, click areas, and focus rings that are broken or missing on desktop. Ensure every interactive element works correctly with mouse and keyboard.

## Context / Why

The app was built mobile-first with touch in mind. Desktop users experience:
- Buttons with no `hover:` state feedback
- Click areas too small or overlapping
- Missing `focus-visible:` rings for keyboard navigation
- Some CTAs that simply don't fire because `onClick` was tied to touch events that don't exist in browser desktop
- `cursor-pointer` missing on clickable elements

## Tasks

### Task 1 — Audit and fix Dashboard interactive elements

<read_first>
- `src/pages/Dashboard.tsx` — main page
- `src/components/dashboard/` — all dashboard sub-components
- `src/components/ui/` — shared UI primitives (Button, Card, etc.)
</read_first>

<action>
Audit every interactive element in the Dashboard and its child components. Apply fixes:

**For every `<button>` element that is missing:**
```tsx
// Add these classes to ALL buttons that lack them:
className="... cursor-pointer hover:opacity-90 active:scale-[0.97] transition-all duration-150 focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-emerald-500 focus-visible:ring-offset-2 focus-visible:ring-offset-zinc-900"
```

**Minimum touch/click target size** — if a button is smaller than 44×44px add:
```tsx
className="... min-h-[44px] min-w-[44px]"
```

**For clickable cards/rows** that use `onClick` but have no cursor or hover:
```tsx
className="... cursor-pointer hover:bg-zinc-800/80 transition-colors duration-150"
```

**Common pattern to apply** — create or update `src/components/ui/Button.tsx` (or the existing button primitive) to a standardized base with all interactive states baked in, so individual usages don't need manual class lists.

Specific targets to fix (check each):
- Dashboard summary cards (income/expense/balance) — should be hoverable if clickable
- Category breakdown rows — hover states
- Quick action buttons — hover + focus rings
- Period filter buttons — active state indicator
</action>

<acceptance_criteria>
- Every `<button>` in `src/components/dashboard/` has `cursor-pointer` in className
- Every `<button>` in `src/components/dashboard/` has a `hover:` variant class
- Every visible button has `min-h-[44px]` OR is documented as intentionally smaller (icon-only inline)
- `npx tsc --noEmit` passes without new errors
</acceptance_criteria>

### Task 3 — Global CSS: ensure scrollbars and desktop-only utilities

<read_first>
- `index.css` (root CSS file)
- `tailwind.config.js`
</read_first>

<action>
Add desktop-specific CSS utilities to `index.css`:

```css
/* Desktop scrollbar styling */
@media (min-width: 768px) {
  ::-webkit-scrollbar {
    width: 6px;
    height: 6px;
  }
  ::-webkit-scrollbar-track {
    background: transparent;
  }
  ::-webkit-scrollbar-thumb {
    background: #3f3f46; /* zinc-700 */
    border-radius: 3px;
  }
  ::-webkit-scrollbar-thumb:hover {
    background: #52525b; /* zinc-600 */
  }
}

/* Ensure pointer cursor for all interactive elements */
button, [role="button"], a {
  cursor: pointer;
}
```

This ensures desktop users get proper scroll indicators and cursor feedback globally, without needing to add `cursor-pointer` to every element.
</action>

<acceptance_criteria>
- `index.css` contains `@media (min-width: 768px)` scrollbar rules
- `index.css` contains `button, [role="button"], a { cursor: pointer; }` rule
- No visual regressions on mobile (scrollbar rules are desktop-only via media query)
</acceptance_criteria>
